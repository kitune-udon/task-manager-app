# AWSリージョン候補整理メモ

## 目的

- 初回 AWS 公開に使うリージョン候補を整理する
- 今回のタスク管理アプリに対して、どのリージョンが運用しやすいかを明確にする
- 最終選択の判断材料を残す

## 前提

- 利用想定は日本国内が中心
- 初回公開対象は MVP 範囲に限定する
- 構成は `RDS -> Elastic Beanstalk -> S3 / CloudFront` を前提にしている
- 初回公開では、まず 1 リージョンでシンプルに立ち上げる方針とする

## 比較観点

- 利用者からの体感遅延
- 構築予定サービスを素直に揃えやすいか
- 今後の運用や調査のしやすさ
- 後続フェーズでの拡張余地
- 初回構築時のシンプルさ

## 候補

### 候補1: `ap-northeast-1` 東京

#### 利点

- 日本国内向けとして体感遅延を抑えやすい
- 初回公開で必要な主要サービスをまとめやすい
- 日本向けの構築事例や情報が多く、調査しやすい
- 後続フェーズで周辺サービスを足すときも運用イメージを保ちやすい

#### 注意点

- 1 リージョン運用なので、災害対策としては別リージョン冗長化が未対応になる
- コスト最適化だけを最優先にする構成ではない

### 候補2: `ap-northeast-3` 大阪

#### 利点

- 日本国内リージョンとして利用できる
- 将来の DR 構成や待機系リージョン候補としては検討しやすい

#### 注意点

- 初回公開のメインリージョンとしては、東京よりも採用理由が弱い
- 運用ノウハウや事例整理は、東京採用時より一段考慮が増える

### 候補3: 海外リージョン

#### 利点

- 既存組織方針や他システムとの統一がある場合は合わせやすい

#### 注意点

- 日本向け MVP の初回公開としては遅延面と説明コストが増えやすい
- 今回の前提では、あえて海外リージョンを選ぶ理由が薄い

## 推奨案

- 初回公開リージョンは `ap-northeast-1` 東京を推奨する

## 推奨理由

- 日本向け MVP の初回公開として、利用者体験と構築の分かりやすさのバランスが最もよい
- backend / DB / 静的配信の構成を 1 リージョン中心で素直に組みやすい
- 後続で大阪リージョンを DR 候補として検討しやすい

## 今回の判断方針

- 2026年4月8日時点では、初回公開は `ap-northeast-1` を第一候補とする
- 特別な組織方針や予算制約がなければ、この方針で固定してよい
- もし「既存 AWS 資産を大阪や海外に寄せたい」前提がある場合は再検討する

## 最終決定

- 2026年4月8日、初回公開リージョンを `ap-northeast-1` 東京で確定した
- 今回の AWS デプロイ作業では、このリージョンを基準に各リソースを作成する

## 補足

- 将来 DR や冗長化を検討する場合は、別リージョンとして大阪を候補にする
