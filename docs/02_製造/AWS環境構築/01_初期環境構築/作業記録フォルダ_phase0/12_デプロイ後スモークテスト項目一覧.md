# デプロイ後スモークテスト項目一覧

## 目的

- AWS へ初回デプロイした直後に、公開継続可否を短時間で判断する
- ローカルで確認済みの観点から、公開後に最低限再確認すべき項目だけを抜き出す
- 障害発生時に、どのレイヤーで異常が起きているかを早く切り分けられるようにする

## 前提

- backend / frontend のデプロイが完了している
- DB migration が適用済みである
- 検証用ユーザーA、必要ならユーザーBが利用できる
- `backend` の環境変数、`frontend` の `VITE_API_BASE_URL` が設定済みである
- コメント、添付、パスワード再発行は初回公開対象外として扱う

## 実施タイミング

- 新規環境へ初回デプロイした直後
- backend / frontend / DB 設定を変更した直後
- ロールバック後の復旧確認時

## 判定ルール

- `OK`: 期待どおりに動作し、次の項目へ進める
- `NG`: 公開継続に支障がある。原因確認またはロールバック判断が必要
- `保留`: 軽微な問題だが、その場で公開可否判断が必要

## 優先度

- `必須`: 公開継続の可否に直結する
- `推奨`: 公開直後に確認したいが、致命傷ではない

## 障害切り分けメモ

- `SMK-01` が NG の場合
  backend の起動失敗、環境変数未設定、DB 接続失敗を優先確認する
- `SMK-02` が NG の場合
  frontend 配信設定、CloudFront/S3 配信、frontend build 成果物を優先確認する
- `SMK-03` から `SMK-12` が NG の場合
  browser の Network、backend ログ、CORS、JWT、DB migration 適用状況を確認する
- `SMK-10` だけ NG の場合
  `/api/users` の疎通、CORS、認証ヘッダ付与を確認する

## 公開継続判断

- `必須` がすべて `OK`
  公開継続可
- `必須` に `NG` が 1 件でもある
  原因調査またはロールバック判断を行う
- `推奨` のみ `NG`
  影響範囲を記録し、公開継続可否を個別判断する
