# S3移行前提DB項目整理メモ

## 目的

- 添付ファイル機能を将来 `S3` 前提で実装する場合の DB 項目を整理する
- 初回 AWS 公開では添付停止のままにしつつ、後続 migration の論点を先に明確にする
- `tasks` テーブルへの安易な列追加を避け、拡張しやすい構成を決める

## 実施日時

- 2026年4月8日

## 現状

- 現行スキーマには `users` と `tasks` のみが存在する
- 添付ファイル用のテーブル、列、API、Entity は未実装
- 初回 AWS 公開では添付ファイル機能を `A. 一時停止` として扱う

## 整理方針

- `tasks` に単一の添付 URL やファイル名列を追加しない
- 将来の複数添付に備えて、添付は独立テーブルで管理する
- 実ファイルは S3、DB にはメタ情報だけを保持する
- 権限制御は task の参照権限に従わせる

## 推奨テーブル案

### テーブル名

- `task_attachments`

### 主な列

| 列名 | 型案 | 必須 | 用途 |
| --- | --- | --- | --- |
| `id` | `BIGSERIAL` | 必須 | 添付ID |
| `task_id` | `BIGINT` | 必須 | 紐づく task |
| `uploaded_by` | `BIGINT` | 必須 | アップロード実行ユーザー |
| `original_file_name` | `VARCHAR(255)` | 必須 | 画面表示用の元ファイル名 |
| `s3_object_key` | `VARCHAR(1024)` | 必須 | S3 上の object key |
| `content_type` | `VARCHAR(255)` | 任意 | MIME type |
| `file_size` | `BIGINT` | 必須 | バイト数 |
| `checksum_sha256` | `VARCHAR(64)` | 任意 | 重複検知や完全性確認用 |
| `status` | `VARCHAR(20)` | 必須 | `ACTIVE` / `DELETED` / `UPLOAD_FAILED` など |
| `created_at` | `TIMESTAMP` | 必須 | 作成日時 |
| `updated_at` | `TIMESTAMP` | 必須 | 更新日時 |
| `deleted_at` | `TIMESTAMP` | 任意 | 論理削除日時 |

## 制約と index 案

- `FOREIGN KEY (task_id) REFERENCES tasks(id)`
- `FOREIGN KEY (uploaded_by) REFERENCES users(id)`
- `INDEX idx_task_attachments_task_id (task_id)`
- `INDEX idx_task_attachments_uploaded_by (uploaded_by)`
- `INDEX idx_task_attachments_status (status)`

## あえて持たない項目

- `s3_url`
  - bucket や domain 変更に弱いため、恒久保存値としては持たない
- `download_url`
  - 署名付き URL は都度発行する前提とし、DB には保持しない
- `tasks.attachment_count`
  - 集計値の同期コストが増えるため、初期設計では持たない

## 実装時の補足

- S3 bucket 名は環境変数または Parameter Store で管理し、DB に固定保存しない前提でよい
- object key は `task/{taskId}/{uuid}-{originalFileName}` のように一意生成する想定
- 削除時は、先に DB を `DELETED` に寄せてから S3 物理削除を行う設計が安全
- 参照 API は task 権限チェックの通過後に添付一覧を返す構成とする

## 今回の結論

- Phase1 では migration 追加までは行わず、DB 項目案の整理までを完了とする
- 実装再開時は `task_attachments` 新設で進める
- `tasks` テーブルへの添付専用列追加は行わない

## 判定

- `1.3-4 S3 移行を見据えた DB 項目を整理する` は対応済み
