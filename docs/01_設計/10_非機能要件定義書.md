## 1. 目的

本書はタスク管理Webサービスにおける非機能要件を定義し、性能・可用性・セキュリティ・運用性・拡張性の観点から設計方針と設計意図を明確化する。  
本ドキュメントはポートフォリオとして、設計判断を説明できる状態を目的とする。

## 2. システム概要

- 構成：React + Spring Boot + PostgreSQL + S3 + AWS
- 主要機能：タスク管理、コメント、添付ファイル、チーム管理、通知
- 利用形態：Webブラウザ（PC中心）

## 3. 性能要件

### 3.1 要件

- ログイン、タスク一覧、詳細表示：3秒以内
- タスク作成 / 更新 / 削除：3秒以内
- コメント投稿：3秒以内

### 3.2 設計方針

- 必要項目のみ取得（DTO最適化）
- N+1問題回避
- 添付ファイルはS3に分離

### 3.3 設計意図

- ユーザー体験の維持
- DB負荷分散
- スケーラビリティ確保

## 4. 可用性

### 4.1 要件

通常利用で安定稼働し、障害時に復旧可能であること。

### 4.2 設計方針

- EC2上でアプリ稼働
- RDSによるDB管理
- CloudWatch Logsで監視

### 4.3 設計意図

- 原因調査を容易にする
- 迅速な復旧を可能にする

## 5. セキュリティ

### 5.1 要件

不正アクセス防止、データ保護。

### 5.2 設計方針

- JWTによる認証
- Spring Securityによるアクセス制御
- パスワードはハッシュ化（BCrypt）
- Service層で認可制御（RBAC）

### 5.3 設計意図

- 認証と認可を分離し責務を明確化
- セキュリティ事故防止

## 6. 運用・監視

### 6.1 要件

障害調査・ログ追跡が可能であること。

### 6.2 設計方針

- CloudWatch Logsにログ集約
- ログ分類（アプリ / 認証 / 例外 / 監査）

### 6.3 設計意図

- 運用効率向上
- 障害対応の迅速化

## 7. 拡張性

### 7.1 要件

機能追加が容易であること。

### 7.2 設計方針

- Controller / Service / Repository分離
- 非同期処理（SQS）拡張前提

### 7.3 設計意図

- 将来的な機能拡張（通知 / バッチ）を容易にする

## 8. バックアップ

### 8.1 要件

データ消失防止。

### 8.2 設計方針

- RDSバックアップ
- S3保存

### 8.3 設計意図

- データ保護と復旧性確保

## 9. 障害シナリオ

- DB接続失敗 → ERRORログ出力 → 再起動
- 認証失敗多発 → WARNログ → 不正アクセス検知
- S3障害 → 添付機能のみ影響

## 10. 設計意図まとめ

本システムは過剰な分散構成を避け、モノリスベースで責務分離を徹底する。  
クラウド（AWS）、セキュリティ、運用を一貫して説明できる構成とする。  
ポートフォリオとして、実務に近い設計判断を示すことを重視する。
