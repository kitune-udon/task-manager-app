<summary>1. 目的</summary>

本書は、task-manager-app における認可の考え方とアクセス制御ルールを定義する。  
JWT による認証と分離し、現行実装および次フェーズ対象機能に対して、どの画面・API・操作が許可されるかを明確化することを目的とする。

次フェーズでは、以下を主要対象とする。

- RBAC ルール棚卸し
- チーム管理
- チーム内ロール管理
- アクティビティログ
- コメント機能拡張
- 添付機能拡張
- 通知一覧
</details>

<summary>3. 基本方針</summary>

- 認証と認可を分離する
- 認証は JWT Bearer 認証で行う
- 認可はチーム所属、チーム内ロール、リソース所有者条件を組み合わせて判定する
- Controller は認可判定を持たず、Service 層へ委譲する
- Repository は DB アクセスのみを担当し、認可ロジックを持たない
- チーム管理機能では、`team_members.role` の `OWNER` / `ADMIN` / `MEMBER` を主要な判定軸とする
- `users.role` はシステム全体ロールとして残すが、チーム管理認可の主要判定軸にはしない
- タスクはチーム所属前提で扱い、対象チームへの所属有無を認可判定の前提とする
- コメント、添付、通知、アクティビティログは、単独判定ではなく親タスクや対象ユーザーとの関係も考慮して判定する
- 論理削除済みリソースは通常APIでは未存在として扱う
- 対象リソースが存在するがチーム非所属または権限不足の場合は、`403 Forbidden` を返却する
</details>

<details>
<summary>4. 認証・認可モデル</summary>

## 4.1 利用者区分

| 区分 | 識別 | 適用 | 説明 |
|---|---|---|---|
| 未認証ユーザー | ANONYMOUS | 現行 / 将来 | ログイン前の利用者 |
| 一般ユーザー | MEMBER | 現行 / 将来 | JWT により認証された通常利用者 |
| チーム所有者 | OWNER | 現行 / 将来 | 対象チームの作成者、またはチーム管理上の最上位権限 |
| チーム管理者 | ADMIN | 現行 / 将来 | 対象チームの管理補助権限 |
| チーム一般メンバー | MEMBER | 現行 / 将来 | 対象チームに所属する一般メンバー |

## 4.2 権限表現

Spring Security 上の権限表現は以下を前提とする。

| 区分 | 権限表現 | 適用 | 説明 |
|---|---|---|---|
| MEMBER | `ROLE_USER` | 現行 / 将来 | 通常の認証済みユーザー |
| ADMIN | `ROLE_ADMIN` | 将来拡張 | システム全体の上位権限を持つユーザー |
| OWNER | `team_members.role = OWNER` | 現行 / 将来 | チーム単位の所有者 |
| ADMIN | `team_members.role = ADMIN` | 現行 / 将来 | チーム単位の管理者 |
| MEMBER | `team_members.role = MEMBER` | 現行 / 将来 | チーム単位の一般メンバー |

※ 現行実装では Spring Security 上は `ROLE_USER` を利用し、チーム機能の認可は `team_members.role` を主軸に判定する。  
※ `ROLE_ADMIN` は将来拡張時のシステム全体管理者を表す設計上の表現として定義する。

## 4.3 リソース所有者条件

認可判定では、ロールに加えて以下の所有者条件を利用する。

| 条件 | 説明 |
|---|---|
| チーム所属者 | `team_members.team_id == targetTeamId` かつ `team_members.user_id == currentUserId` |
| チーム管理者 | 対象チームにおける `team_members.role` が `OWNER` または `ADMIN` |
| タスク作成者 | `task.createdBy.id == currentUserId` |
| タスク担当者 | `task.assignedUser.id == currentUserId` |
| コメント投稿者 | `comment.createdBy.id == currentUserId` |
| 添付登録者 | `attachment.createdBy.id == currentUserId` |
| 通知受信者 | `notification.userId == currentUserId` |
| 自分関連アクティビティ | 対象イベントが currentUserId と関連すること |

## 4.4 認可の基本ルール

| 観点 | ルール |
|---|---|
| 未認証アクセス | 常に拒否 |
| チーム作成 | 認証済みユーザーであれば許可する |
| チーム参照 | 対象チームに所属している場合に許可する |
| チームメンバー管理 | `OWNER` または `ADMIN` を基本とし、ロール変更は `OWNER` のみ許可する |
| タスク参照 | 対象タスクの所属チームに所属している場合に許可する |
| タスク更新 | 対象チーム所属に加え、作成者、担当者、`OWNER`、`ADMIN` のいずれかで許可する |
| タスク削除 | 対象チーム所属に加え、作成者、`OWNER`、`ADMIN` のいずれかで許可する |
| コメント操作 | 親タスクのチーム所属を前提とし、更新 / 削除は投稿者またはチーム管理者に許可する |
| 添付操作 | 親タスクのチーム所属を前提とし、アップロードはタスク更新権限、削除は登録者またはチーム管理者に許可する |
| 通知参照 | 自分宛のみ許可する |
| アクティビティログ参照 | 対象タスクの所属チームに所属している場合に許可する |
</details>

## 5.3 共通ルール

- 上記以外の API は認証必須とする
- 認証情報がない場合は `401 Unauthorized` を返却する
- JWT が不正または期限切れの場合も `401 Unauthorized` を返却する
- 認証済みであっても、認可条件を満たさない場合は `403 Forbidden` を返却する
- 対象リソースが存在しない、または論理削除済みの場合は `404 Not Found` を返却する

<details>
<summary>6. 認可判定ルール</summary>

## 6.1 判定手順

認可判定は、次の順序で行う。

1. 認証済みであることを確認する
2. 対象データを取得する
3. 対象データが論理削除済みでないことを確認する
4. 現在のログインユーザー ID を取得する
5. 対象チームへの所属有無を判定する
6. 操作種別に応じて、チーム内ロール条件または所有者条件を判定する
7. 条件を満たさない場合は認可エラーを返す
8. 担当中タスクを持つメンバー削除禁止など、認可以外の業務制約を判定する

## 6.2 タスク参照ルール

タスク参照は、以下の条件で許可する。

- 対象タスクが存在すること
- 対象タスクが論理削除済みでないこと
- ログインユーザーが対象タスクの所属チームに所属していること

タスク作成者または担当者であっても、現在対象チームに所属していない場合は参照不可とする。

## 6.3 タスク更新ルール

タスク更新は、以下の条件で許可する。

- 対象タスクが存在すること
- 対象タスクが論理削除済みでないこと
- ログインユーザーが対象タスクの所属チームに所属していること
- 以下のいずれかを満たすこと
  - タスク作成者
  - タスク担当者
  - 対象チームの `OWNER`
  - 対象チームの `ADMIN`

## 6.4 タスク削除ルール

タスク削除は、以下の条件で許可する。

- 対象タスクが存在すること
- 対象タスクが論理削除済みでないこと
- ログインユーザーが対象タスクの所属チームに所属していること
- 以下のいずれかを満たすこと
  - タスク作成者
  - 対象チームの `OWNER`
  - 対象チームの `ADMIN`

担当者であるだけでは削除不可とする。

## 6.5 タスク複製ルール

タスク複製は将来拡張想定とし、本フェーズでは実装対象外とする。実装する場合は、元タスクの参照権限を持つ場合に許可する。

## 6.6 ユーザー一覧取得ルール

担当者候補一覧取得は、認証済みユーザーであれば許可する。

タスク作成・更新で担当者を指定する場合、対象ユーザーがタスク所属チームのメンバーであることを別途検証する。

## 6.7 コメント参照 / 投稿ルール

コメント一覧取得およびコメント投稿は、以下の条件で許可する。

- 親タスクが存在すること
- 親タスクが論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること

コメント投稿者であっても、現在対象チームに所属していない場合は参照・投稿不可とする。

## 6.8 コメント更新 / 削除ルール

コメント更新および削除は、以下の条件で許可する。

- 対象コメントが存在すること
- 対象コメントが論理削除済みでないこと
- 親タスクが存在し、論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること
- 以下のいずれかを満たすこと
  - コメント投稿者本人
  - 対象チームの `OWNER`
  - 対象チームの `ADMIN`

## 6.9 添付参照 / ダウンロードルール

添付一覧取得およびダウンロードは、以下の条件で許可する。

- 親タスクまたは対象添付が存在すること
- 親タスクまたは対象添付が論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること

添付登録者であっても、現在対象チームに所属していない場合は参照・ダウンロード不可とする。

## 6.10 添付アップロードルール

添付アップロードは、以下の条件で許可する。

- 親タスクが存在すること
- 親タスクが論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること
- ログインユーザーが対象タスクを更新可能であること

## 6.11 添付削除ルール

添付削除は、以下の条件で許可する。

- 対象添付が存在すること
- 対象添付が論理削除済みでないこと
- 親タスクが存在し、論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること
- 以下のいずれかを満たすこと
  - 添付登録者本人
  - 対象チームの `OWNER`
  - 対象チームの `ADMIN`

## 6.12 通知一覧取得 / 未読件数取得 / 既読化ルール

通知一覧取得、未読件数取得、既読化、一括既読化は、自分宛の通知に対してのみ許可する。

通知自体の認可にチーム内ロールは直接利用しない。通知から関連タスクへ遷移する場合は、遷移先タスクの参照認可で判定する。

## 6.13 タスク別履歴参照ルール

タスク別履歴一覧取得は、以下の条件で許可する。

- 親タスクが存在すること
- 親タスクが論理削除済みでないこと
- ログインユーザーが親タスクの所属チームに所属していること

## 6.14 チーム参照ルール

チーム詳細取得、チーム所属メンバー一覧取得は、以下の条件で許可する。

- 対象チームが存在すること
- ログインユーザーが対象チームに所属していること

所属チーム一覧取得は、自分が所属するチームのみ返却するため、個別チームの認可判定は不要とする。

## 6.15 チームメンバー管理ルール

チームメンバー管理は、以下の条件で許可する。

| 操作 | 許可条件 | 補足 |
|---|---|---|
| 追加候補ユーザー一覧取得 | 対象チームの `OWNER` または `ADMIN` | `MEMBER` は不可 |
| メンバー追加 | 対象チームの `OWNER` または `ADMIN` | 追加時ロールは `ADMIN` または `MEMBER` |
| ロール変更 | 対象チームの `OWNER` | `OWNER` のロール変更不可 |
| メンバー削除 | 対象チームの `OWNER` または `ADMIN` | `OWNER` 削除不可、担当中タスクがあるメンバーは削除不可 |

`ADMIN` の自己削除は許可する。削除後は対象チームへの所属が失効するため、画面側では所属チーム一覧へ遷移する。
</details>

<summary>8. 実装方針</summary>

## 8.1 レイヤ責務

| レイヤ | 役割 | 備考 |
|---|---|---|
| SecurityFilterChain | 公開/保護 API の一次制御 | 未認証アクセスを遮断する |
| JwtAuthenticationFilter | JWT を検証し SecurityContext を設定 | 不正・期限切れは 401 を返す |
| CurrentUserProvider | 現在ユーザー情報を取得 | currentUserId の取得に利用 |
| Controller | リクエスト受付と Service 呼び出し | 認可ロジックは持たない |
| Service | 対象データ取得、チーム所属判定、チーム内ロール判定、所有者条件判定、認可エラー送出 | 認可判定の中心 |
| Repository | DB アクセス | 認可ロジックは持たない |

## 8.2 現行実装の認可ロジック

|---|---|

| `authorizeTeamMemberRemove` | 対象チーム所属かつ `OWNER` / `ADMIN` であることを確認する |
| `authorizeTaskListInTeam` | 指定チームへの所属を確認する |
| `authorizeTaskCreateInTeam` | 指定チームへの所属を確認する |

| 想定メソッド | 内容 |
|---|---|
| `authorizeAttachmentUploadInTeam` | 親タスクの更新権限を前提に添付アップロード可否を判定する |

## 8.4 実装上の補足

- コメント / 添付の認可は、単独テーブルだけでなく親タスク権限も合わせて判定する
- 通知 / アクティビティログは、参照系の機能を先行し、作成処理は業務イベント発火側で責務分離する
- チーム認可では、`users.role` ではなく `team_members.role` を主軸とする
- チーム所属判定と所有者判定を Controller に分散させず、Service / AuthorizationService へ集約する
- 今後 `@PreAuthorize` を導入する場合でも、業務条件が複雑な箇所は Service 層判定を優先する

<summary>9. エラー設計との対応</summary>

| エラーコード | HTTP | 意味 | 主な発生条件 |
|---|---:|---|---|
| `ERR-AUTH-001` | 401 | 認証が必要です | 未認証アクセス |
| `ERR-AUTH-003` | 401 | トークンが不正です | JWT 不正 |
| `ERR-AUTH-004` | 401 | セッションの有効期限が切れています | JWT 期限切れ |
| `ERR-AUTH-005` | 403 | 操作権限がありません | 共通的な権限不足 |
| `ERR-TEAM-003` | 403 | このチームにアクセスする権限がありません | 非所属ユーザーがチーム参照 |
| `ERR-TEAM-004` | 404 | 対象のチームが存在しません | teamId 不存在 |
| `ERR-TEAM-MEMBER-002` | 403 | メンバー追加権限がありません | `OWNER` / `ADMIN` 以外がメンバー追加または追加候補取得 |
| `ERR-TEAM-MEMBER-005` | 403 | ロール変更権限がありません | `OWNER` 以外がロール変更 |
| `ERR-TEAM-MEMBER-006` | 403 | OWNERのロールは変更できません | OWNER のロール変更操作 |
| `ERR-TEAM-MEMBER-008` | 403 | メンバー削除権限がありません | `OWNER` / `ADMIN` 以外がメンバー削除 |
| `ERR-TEAM-MEMBER-009` | 403 | OWNERは削除できません | OWNER 削除操作 |
| `ERR-TEAM-MEMBER-010` | 409 | 担当中のタスクがあるため、このメンバーを削除できません | 担当中タスクを持つメンバー削除 |
| `ERR-TASK-005` | 403 | タスク更新権限がありません | タスク更新時、作成者 / 担当者 / `OWNER` / `ADMIN` 以外 |
| `ERR-TASK-006` | 403 | タスク削除権限がありません | タスク削除時、作成者 / `OWNER` / `ADMIN` 以外 |
| `ERR-TASK-008` | 400 | タスクを作成するチームを指定してください | タスク作成時の teamId 未指定 |
| `ERR-TASK-009` | 403 | 指定されたチームの操作権限がありません | 指定チームまたはタスク所属チームに非所属 |
| `ERR-TASK-010` | 400 | 担当者はチームメンバーから選択してください | チーム外ユーザーを担当者に指定 |
| `ERR-COMMENT-004` | 403 | コメント更新権限がありません | コメント更新時、投稿者 / `OWNER` / `ADMIN` 以外 |
| `ERR-COMMENT-005` | 403 | コメント削除権限がありません | コメント削除時、投稿者 / `OWNER` / `ADMIN` 以外 |
| `ERR-FILE-003` | 403 | 添付アップロード権限がありません | 親タスク更新権限がない |
| `ERR-FILE-004` | 403 | 添付削除権限がありません | 添付削除時、登録者 / `OWNER` / `ADMIN` 以外 |
| `ERR-NOTIFY-002` | 403 | 通知参照権限がありません | 他人宛通知へアクセスした |
| `ERR-ACTIVITY-002` | 403 | アクティビティログ参照権限がありません | 対象タスク参照権限がない |

### 9.1 補足

- 認可エラーは `403 Forbidden` で返却する
- 認証エラーと認可エラーは分離して扱う
- 対象リソースが存在しない場合、または論理削除済みの場合は `404 Not Found` を返却する
- 担当中タスクを持つメンバー削除禁止など、権限不足ではない業務制約は `409 Conflict` を返却する
- エラーレスポンス形式は `ErrorResponse` に統一する
- チーム認可では、`users.role` ではなく `team_members.role` を判定軸とする
- コメント / 添付 / 通知 / アクティビティログ追加時は、エラー設計書側も合わせて改訂する
</details>

<summary>10. 将来拡張時の観点</summary>

次フェーズでは未対応だが、将来的に以下の認可ルール追加が想定される。

| 機能 | 想定する追加条件 |
|---|---|
| チーム削除 | OWNER のみ実行可能。削除時のタスク・メンバー・履歴の扱いを別途定義する |
| メンバー招待 | 招待作成・再送・失効は `OWNER` / `ADMIN` のみ |
| 招待リンク | リンク作成・無効化・有効期限管理は `OWNER` / `ADMIN` のみ |
| チーム単位の詳細権限設定 | 操作単位の細かな権限をチームごとに設定可能にする |

| 通知配信 | メール通知や Push 通知は配信対象の明確化が必要 |
| 監査強化 | アクティビティログの参照範囲と保存期間の詳細設計 |

### 10.1 拡張方針

- まずはチーム所属、チーム内ロール、所有者条件で説明可能な構成を優先する
- `ROLE_ADMIN` を導入する場合は、チーム単位権限とは別のシステム管理権限として定義する
- 権限種別を増やす前に、所有者条件と参照範囲で表現できるかを優先検討する
- 将来の機能追加時は、本書とエラー設計書を合わせて改訂する
</details>

<summary>11. 設計意図</summary>

本システムでは、認証と認可を分離し、認可判定を Service 層へ寄せることで責務を明確にしている。  
また、単純なロール判定だけではなく、対象リソースとの関係性を条件に加えることで、実務上自然なアクセス制御を実現している。  
チーム管理機能では、システム全体ロールではなく対象チームへの所属有無とチーム内ロールを主軸にすることで、複数チーム所属時のアクセス境界を明確にする。  
コメント、添付、通知、アクティビティログでは、個別機能ごとに判定を分散させるのではなく、親タスクの所属チームや受信者本人条件を共通ルールとして整理し、説明しやすく保守しやすい構成を目指す。
</details>

<summary>12. 備考</summary>

- 現行実装で付与される Spring Security 上の権限は `ROLE_USER` のみである
- `ROLE_ADMIN` は本書上の将来拡張向け設計表現であり、チーム内 `ADMIN` とは別概念として扱う
- チーム管理機能の認可は `team_members.role` の `OWNER` / `ADMIN` / `MEMBER` を主軸にする
- コメント、添付、通知、アクティビティログに関する認可は、親タスクの所属チームまたは通知受信者本人条件に基づいて判定する
- チーム削除、招待、招待リンク、チーム単位の詳細権限設定は将来拡張として扱う
- 削除済みタスクは未存在として扱い、通常APIでは参照不可とする
</details>
