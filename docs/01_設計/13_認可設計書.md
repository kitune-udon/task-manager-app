<summary>1. 目的</summary>

本書は、task-manager-app における認可の考え方とアクセス制御ルールを定義する。  
JWT による認証と分離し、現行実装および次フェーズ対象機能に対して、どの画面・API・操作が許可されるかを明確化することを目的とする。

次フェーズでは、以下を主要対象とする。

- RBAC ルール棚卸し
- アクティビティログ
- コメント機能拡張
- 添付機能拡張
- 通知一覧
</details>

<summary>3. 基本方針</summary>

- 認証と認可を分離する
- 認証は JWT Bearer 認証で行う
- 認可は RBAC とリソース所有者条件を組み合わせて判定する
- Controller は認可判定を持たず、Service 層へ委譲する
- Repository は DB アクセスのみを担当し、認可ロジックを持たない
- RBAC は上位権限の表現に用い、細かな操作可否は対象データとの関係性で補完する
- コメント、添付、通知、アクティビティログは、単独判定ではなく親タスクや対象ユーザーとの関係も考慮して判定する
- 現行実装では Spring Security 上の権限は `ROLE_USER` のみを利用する
- `ADMIN` は将来拡張時に導入する上位権限として扱い、現行実装とは分けて記載する
- 将来的にチーム管理機能を導入する場合、`ADMIN` の適用範囲をチーム単位へ具体化する
</details>

<summary>4. 認証・認可モデル</summary>

## 4.1 利用者区分

| 区分 | 識別 | 適用 | 説明 |
|---|---|---|---|
| 未認証ユーザー | ANONYMOUS | 現行 / 将来 | ログイン前の利用者 |
| 一般ユーザー | MEMBER | 現行 / 将来 | JWT により認証された通常利用者 |
| 管理者ユーザー | ADMIN | 将来拡張 | 上位権限を持つ認証済み利用者 |

## 4.2 権限表現

Spring Security 上の権限表現は以下を前提とする。

| 区分 | 権限表現 | 適用 | 説明 |
|---|---|---|---|
| MEMBER | `ROLE_USER` | 現行 / 将来 | 通常の認証済みユーザー |
| ADMIN | `ROLE_ADMIN` | 将来拡張 | 上位権限を持つユーザー |

※ 現行実装では `ROLE_USER` のみ付与されている。  
※ `ROLE_ADMIN` は次フェーズ以降の設計上の表現として定義する。

## 4.3 リソース所有者条件

認可判定では、ロールに加えて以下の所有者条件を利用する。

| 条件 | 説明 |
|---|---|
| タスク作成者 | `task.createdBy.id == currentUserId` |
| タスク担当者 | `task.assignedUser.id == currentUserId` |
| コメント投稿者 | `comment.createdBy.id == currentUserId` |
| 添付登録者 | `attachment.createdBy.id == currentUserId` |
| 通知受信者 | `notification.userId == currentUserId` |
| 自分関連アクティビティ | 対象イベントが currentUserId と関連すること |

## 4.4 認可の基本ルール

| 観点 | ルール |
|---|---|
| 未認証アクセス | 常に拒否 |
| MEMBER（現行実装） | 所有者条件と親タスク参照 / 更新権限で判定する |
| ADMIN（将来拡張） | 一部操作で所有者条件を上書き可能とする |
| コメント操作 | 親タスク参照権限を前提とする |
| 添付操作 | 親タスク参照 / 更新権限を前提とする |
| 通知参照 | 自分宛のみ許可する |
| アクティビティログ参照 | 対象タスクの参照権限を持つ場合に許可する |
</details>

<summary>6. 認可判定ルール</summary>

## 6.1 判定手順

認可判定は、次の順序で行う。

1. 認証済みであることを確認する
2. 対象データを取得する
3. 現在のログインユーザー ID とロールを取得する
4. 将来拡張として `ADMIN` が導入されている場合は、ADMIN かどうかを判定する
5. 操作種別に応じて、所有者条件または参照可能条件を判定する
6. 条件を満たさない場合は認可エラーを返す

## 6.2 タスク参照ルール

タスク参照は、以下の条件で許可する。

- 現行実装: タスク作成者またはタスク担当者
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

## 6.3 タスク更新ルール

タスク更新は、以下の条件で許可する。

- 現行実装: タスク作成者またはタスク担当者
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

## 6.4 タスク削除ルール

タスク削除は、以下の条件で許可する。

- 現行実装: タスク作成者
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

## 6.5 タスク複製ルール

タスク複製は将来拡張想定とし、本フェーズでは実装対象外とする。実装する場合は、元タスクの参照権限を持つ場合に許可する。

## 6.6 ユーザー一覧取得ルール

担当者候補一覧取得は、認証済みユーザーであれば許可する。

## 6.7 コメント参照 / 投稿ルール

コメント一覧取得およびコメント投稿は、対象タスクの参照権限を持つ場合に許可する。

## 6.8 コメント更新 / 削除ルール

コメント更新および削除は、以下の条件で許可する。

- 現行実装: コメント投稿者本人
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

## 6.9 添付参照 / ダウンロードルール

添付一覧取得およびダウンロードは、対象タスクの参照権限を持つ場合に許可する。

## 6.10 添付アップロードルール

添付アップロードは、対象タスクの更新権限を持つ場合に許可する。

## 6.11 添付削除ルール

添付削除は、以下の条件で許可する。

- 現行実装: 添付登録者本人
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

## 6.12 通知一覧取得 / 未読件数取得 / 既読化ルール

## 6.13 タスク別履歴参照ルール

- 現行実装: タスク作成者またはタスク担当者
- 将来拡張: `ADMIN` 導入時は ADMIN も許可

<summary>8. 実装方針</summary>

## 8.1 レイヤ責務

| レイヤ | 役割 | 備考 |
|---|---|---|
| SecurityFilterChain | 公開/保護 API の一次制御 | 未認証アクセスを遮断する |
| JwtAuthenticationFilter | JWT を検証し SecurityContext を設定 | 不正・期限切れは 401 を返す |
| CurrentUserProvider | 現在ユーザー情報を取得 | currentUserId / role の取得に利用 |
| Controller | リクエスト受付と Service 呼び出し | 認可ロジックは持たない |
| Service | 対象データ取得、ロール判定、所有者条件判定、認可エラー送出 | 認可判定の中心 |
| Repository | DB アクセス | 認可ロジックは持たない |

## 8.2 現行実装の認可ロジック

| メソッド | 内容 |
|---|---|
| `authorizeTaskView` | 作成者または担当者のみ参照可 |
| `authorizeTaskUpdate` | 作成者または担当者のみ更新可 |
| `authorizeTaskDelete` | 作成者のみ削除可 |
| `canViewTask` | `isTaskCreator` または `isTaskAssignee` |
| `canUpdateTask` | `canViewTask` と同条件 |
| `canDeleteTask` | `isTaskCreator` のみ |

## 8.3 次フェーズで追加する認可ロジック

| 想定メソッド | 内容 |
|---|---|
| `isAdmin` | 将来拡張として管理者権限を持つか判定する |
| `authorizeCommentView` | 親タスク参照権限を前提にコメント参照可 |
| `authorizeCommentCreate` | 親タスク参照権限を前提にコメント投稿可 |
| `authorizeCommentUpdate` | コメント投稿者本人または ADMIN のみ更新可 |
| `authorizeCommentDelete` | コメント投稿者本人または ADMIN のみ削除可 |
| `authorizeAttachmentList` | 親タスク参照権限を前提に添付一覧取得可 |
| `authorizeAttachmentUpload` | 親タスク更新権限を前提に添付アップロード可 |
| `authorizeAttachmentDownload` | 親タスク参照権限を前提に添付ダウンロード可 |
| `authorizeAttachmentDelete` | 添付登録者本人または ADMIN のみ削除可 |
| `authorizeNotificationAccess` | 自分宛通知のみ参照・既読化可 |
| `authorizeTaskActivityView` | 対象タスク参照権限を前提にタスク別履歴一覧取得可 |

## 8.4 実装上の補足

- コメント / 添付の認可は、単独テーブルだけでなく親タスク権限も合わせて判定する
- 通知 / アクティビティログは、参照系の機能を先行し、作成処理は業務イベント発火側で責務分離する
- 将来 `ADMIN` を導入する場合でも、ADMIN 判定を Controller へ分散させず、共通化された Service / AuthorizationService へ集約する
- 今後 `@PreAuthorize` を導入する場合でも、業務条件が複雑な箇所は Service 層判定を優先する
</details>

<summary>9. エラー設計との対応</summary>

| エラーコード | HTTP | 意味 | 主な発生条件 |
|---|---:|---|---|
| `ERR-AUTH-001` | 401 | 認証が必要です | 未認証アクセス |
| `ERR-AUTH-003` | 401 | トークンが不正です | JWT 不正 |
| `ERR-AUTH-004` | 401 | セッションの有効期限が切れています | JWT 期限切れ |
| `ERR-AUTH-005` | 403 | 操作権限がありません | 共通的な権限不足 |
| `ERR-TASK-005` | 403 | タスク更新権限がありません | タスク更新時、作成者/担当者/ADMIN 以外 |
| `ERR-TASK-006` | 403 | タスク削除権限がありません | タスク削除時、作成者/ADMIN 以外 |
| `ERR-COMMENT-004` | 403 | コメント更新権限がありません | コメント更新時、投稿者/ADMIN 以外 |
| `ERR-COMMENT-005` | 403 | コメント削除権限がありません | コメント削除時、投稿者/ADMIN 以外 |
| `ERR-FILE-003` | 403 | 添付アップロード権限がありません | 親タスク更新権限がない |
| `ERR-FILE-004` | 403 | 添付削除権限がありません | 添付削除時、登録者/ADMIN 以外 |
| `ERR-NOTIFY-002` | 403 | 通知参照権限がありません | 他人宛通知へアクセスした |
| `ERR-ACTIVITY-002` | 403 | アクティビティログ参照権限がありません | 対象タスク参照権限がない |

### 9.1 補足

- 認可エラーは `403 Forbidden` で返却する
- 認証エラーと認可エラーは分離して扱う
- エラーレスポンス形式は `ErrorResponse` に統一する
- 現行実装では `ADMIN` 未導入のため、実際の判定は MEMBER と所有者条件を前提とする
- コメント / 添付 / 通知 / アクティビティログ追加時は、エラー設計書側も合わせて改訂する
</details>

| 機能 | 想定する追加条件 |

### 10.1 拡張方針

- 権限種別を増やす前に、所有者条件と参照範囲で表現できるかを優先検討する
- 将来の機能追加時は、本書とエラー設計書を合わせて改訂する

<details>
<summary>11. 設計意図</summary>

本システムでは、認証と認可を分離し、認可判定を Service 層へ寄せることで責務を明確にしている。  
また、単純なロール判定だけではなく、対象リソースとの関係性を条件に加えることで、実務上自然なアクセス制御を実現している。  
次フェーズでは、コメント、添付、通知、アクティビティログを追加するにあたり、個別機能ごとに判定を分散させるのではなく、RBAC と所有者条件を共通ルールとして整理し、説明しやすく保守しやすい構成を目指す。
</details>
