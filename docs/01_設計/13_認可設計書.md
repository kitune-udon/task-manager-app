## 1. 目的

本書は、task-manager-app における認可の考え方とアクセス制御ルールを定義する。  
JWT による認証と分離し、現行システムでどの画面・API・操作が許可されるかを明確化することを目的とする。

## 2. 適用範囲

本書の対象は、現行実装に含まれる以下の機能とする。

- 認証
  - 新規登録
  - ログイン
  - ログアウト
- タスク
  - 一覧取得
  - 詳細取得
  - 作成
  - 更新
  - 削除
- ユーザー
  - 担当者候補一覧取得

以下は現時点では対象外とし、将来拡張として扱う。

- コメント
- 添付ファイル
- チーム管理
- 通知
- アクティビティログ
- 管理者ロールによる権限制御

## 3. 基本方針

- 認証と認可を分離する
- 認証は JWT Bearer 認証で行う
- 認可は「ログイン済みかどうか」に加え、タスクとの関係性で判定する
- Controller は認可判定を持たず、Service 層へ委譲する
- Repository は DB アクセスのみを担当し、認可ロジックを持たない
- 現行実装では、管理者ロールによる権限制御は導入しない
- 現行実装では、タスク認可は「作成者」「担当者」の条件を組み合わせて判定する

## 4. 認証・認可モデル

## 4.1 利用者区分

| 区分 | 識別 | 説明 |
|---|---|---|
| 未認証ユーザー | ANONYMOUS | ログイン前の利用者 |
| 認証済みユーザー | AUTHENTICATED | JWT により認証された利用者 |

## 4.2 認証済みユーザーの権限表現

現行実装では、認証済みユーザーは Spring Security 上で `ROLE_USER` を持つ。  
ただし、業務上の操作可否はロールだけでなく、対象タスクとの関係性によって決定する。

## 4.3 リソース所有者条件

タスクに対する認可判定では、以下の条件を利用する。

| 条件 | 説明 |
|---|---|
| 作成者 | `task.createdBy.id == currentUserId` |
| 担当者 | `task.assignedUser.id == currentUserId` |

## 5. 公開エンドポイントと保護エンドポイント

## 5.1 公開エンドポイント

| API | 用途 |
|---|---|
| `POST /api/auth/register` | 新規登録 |
| `POST /api/auth/login` | ログイン |
| `GET /actuator/health` | ヘルスチェック |
| `/error` | エラー画面 |

## 5.2 保護エンドポイント

| API | 用途 |
|---|---|
| `GET /api/tasks` | タスク一覧取得 |
| `GET /api/tasks/{id}` | タスク詳細取得 |
| `POST /api/tasks` | タスク作成 |
| `PUT /api/tasks/{id}` | タスク更新 |
| `DELETE /api/tasks/{id}` | タスク削除 |
| `GET /api/users` | 担当者候補一覧取得 |

## 5.3 共通ルール

- 上記以外の API は認証必須とする
- 認証情報がない場合は `401 Unauthorized` を返却する
- JWT が不正または期限切れの場合も `401 Unauthorized` を返却する

## 6. 認可判定ルール

## 6.1 判定手順

認可判定は、次の順序で行う。

1. 認証済みであることを確認する
2. 対象データを取得する
3. 現在のログインユーザー ID を取得する
4. 操作種別に応じて、作成者・担当者条件を判定する
5. 条件を満たさない場合は認可エラーを返す

## 6.2 タスク参照ルール

タスク参照は、以下のいずれかを満たす場合に許可する。

- タスク作成者
- タスク担当者

## 6.3 タスク更新ルール

タスク更新は、以下のいずれかを満たす場合に許可する。

- タスク作成者
- タスク担当者

## 6.4 タスク削除ルール

タスク削除は、以下を満たす場合のみ許可する。

- タスク作成者

## 6.5 ユーザー一覧取得ルール

担当者候補一覧取得は、認証済みユーザーであれば許可する。

## 7. アクセス制御マトリクス

## 7.1 認証系

| 機能 | ANONYMOUS | AUTHENTICATED | 条件 / 補足 |
|---|---|---|---|
| 新規登録 | ○ | - | 未認証時のみ利用想定 |
| ログイン | ○ | - | 未認証時のみ利用想定 |
| ログアウト | - | ○ | 認証済みユーザーのみ |

## 7.2 タスク系

| 機能 | ANONYMOUS | AUTHENTICATED | 条件 / 補足 |
|---|---|---|---|
| タスク一覧取得 | - | ○ | 自分が参照可能なタスクのみ |
| タスク詳細取得 | - | △ | 作成者または担当者のみ |
| タスク作成 | - | ○ | 認証済みユーザー |
| タスク更新 | - | △ | 作成者または担当者のみ |
| タスク削除 | - | △ | 作成者のみ |

## 7.3 ユーザー系

| 機能 | ANONYMOUS | AUTHENTICATED | 条件 / 補足 |
|---|---|---|---|
| 担当者候補一覧取得 | - | ○ | 認証済みユーザー |

## 8. 実装方針

## 8.1 レイヤ責務

| レイヤ | 役割 | 備考 |
|---|---|---|
| SecurityFilterChain | 公開/保護 API の一次制御 | 未認証アクセスを遮断する |
| JwtAuthenticationFilter | JWT を検証し SecurityContext を設定 | 不正・期限切れは 401 を返す |
| CurrentUserProvider | 現在ユーザー情報を取得 | currentUserId の取得に利用 |
| Controller | リクエスト受付と Service 呼び出し | 認可ロジックは持たない |
| Service | 対象データ取得、所有者条件判定、認可エラー送出 | 認可判定の中心 |
| Repository | DB アクセス | 認可ロジックは持たない |

## 8.2 実装済み認可ロジック

| メソッド | 内容 |
|---|---|
| `authorizeTaskView` | 作成者または担当者のみ参照可 |
| `authorizeTaskUpdate` | 作成者または担当者のみ更新可 |
| `authorizeTaskDelete` | 作成者のみ削除可 |
| `canViewTask` | `isTaskCreator` または `isTaskAssignee` |
| `canUpdateTask` | `canViewTask` と同条件 |
| `canDeleteTask` | `isTaskCreator` のみ |

## 9. エラー設計との対応

| エラーコード | HTTP | 意味 | 主な発生条件 |
|---|---:|---|---|
| `ERR-AUTH-001` | 401 | 認証が必要です | 未認証アクセス |
| `ERR-AUTH-003` | 401 | トークンが不正です | JWT 不正 |
| `ERR-AUTH-004` | 401 | セッションの有効期限が切れています | JWT 期限切れ |
| `ERR-AUTH-005` | 403 | 操作権限がありません | 共通的な権限不足 |
| `ERR-TASK-005` | 403 | タスク更新権限がありません | タスク更新時、作成者/担当者以外 |
| `ERR-TASK-006` | 403 | タスク削除権限がありません | タスク削除時、作成者以外 |

### 9.1 補足

- 認可エラーは `403 Forbidden` で返却する
- 認証エラーと認可エラーは分離して扱う
- エラーレスポンス形式は `ErrorResponse` に統一する

## 10. 将来拡張時の観点

現行実装では未対応だが、将来的に以下の認可ルール追加が想定される。

| 機能 | 想定する追加条件 |
|---|---|
| コメント | 投稿者本人のみ更新・削除可 |
| 添付ファイル | 添付登録者本人のみ削除可 |
| チーム管理 | チーム管理者のみメンバー追加・削除・ロール変更可 |
| 通知 | 自分宛の通知のみ参照可 |
| アクティビティログ | 自分関連または所属範囲のみ参照可 |
| 管理者ロール | `ADMIN` による上位権限付与 |

### 10.1 拡張方針

- まずは `ROLE_USER` を前提とした現行実装を維持する
- 管理者ロール導入時は、User/Team モデルと認可判定の両方を拡張する
- コメント、添付、チーム機能の追加時に、本書へ対象ルールを追記する

## 11. 設計意図

本システムでは、認証と認可を分離し、認可判定を Service 層へ寄せることで責務を明確にしている。  
また、単純なロール判定だけではなく、タスクとの関係性を条件に加えることで、実務上自然なアクセス制御を実現している。  
現行実装では過剰なロール設計を避け、まずは「認証済みユーザー + 所有者条件」に絞ることで、シンプルで説明しやすい構成としている。

## 12. 備考

- 現行実装で付与される権限は `ROLE_USER` のみである
- `ADMIN`、チーム管理、コメント、添付に関する認可は未実装である
- 今後機能追加時は、本書とエラー設計書を合わせて改訂する
