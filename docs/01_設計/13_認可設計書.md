## 1. 目的

本書はタスク管理Webサービスにおける認可（Authorization）の設計を定義する。  
JWT による認証と分離し、ロールおよびリソース所有者条件に基づくアクセス制御を明確化する。

## 2. 基本方針

- 認証（Authentication）と認可（Authorization）を分離する
- 認可は RBAC（Role Based Access Control）をベースとする
- RBAC だけでは不足するため、所有者ベース制御（本人・作成者・担当者・チーム管理者）を組み合わせる
- 最終的な認可判定は Service 層で実施し、Controller はリクエスト受領と委譲に責務を限定する
- 2026年4月8日時点の実装は第1段階として、タスク認可を作成者・担当者ベースまでとし、チーム ADMIN 条件は後続対応とする

## 3. ロール定義

| ロール | 識別子 | 説明 |
|---|---|---|
| 未認証ユーザー | ANONYMOUS | ログイン前の利用者。公開画面のみアクセス可能。 |
| 一般ユーザー | MEMBER | 通常ユーザー。タスク参照・作成・本人コメント操作などが可能。 |
| チーム管理者 | ADMIN | チーム内のメンバー管理、ロール変更、広い操作権限を持つ。 |

## 4. 認可判定の考え方

1. ログインしているか
2. 対象ロールを持っているか
3. 対象リソースとの関係（作成者・担当者・投稿者・所属チーム管理者）を満たすか

- 判定方式: RBAC + 所有者ベース制御

## 5. アクセス制御マトリクス

### 5.1 認証系マトリクス

| 機能 | 未認証 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|---|
| 新規登録 | ○ | - | - | 未認証時のみ利用可 |
| ログイン | ○ | - | - | 未認証時のみ利用可 |
| ログアウト | - | ○ | ○ | ログイン済みユーザーのみ |

### 5.2 タスク系マトリクス

| 機能 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|
| タスク一覧取得 | ○ | ○ | 自分が参照可能な範囲 |
| タスク詳細取得 | ○ | ○ | 対象タスク参照権限あり |
| タスク作成 | ○ | ○ | ログイン済みユーザー |
| タスク更新 | △ | ○ | MEMBER は作成者 / 担当者 / 許可対象のみ |
| タスク削除 | △ | ○ | MEMBER は作成者のみ、ADMIN はチーム内可 |
| タスク複製 | ○ | ○ | 参照できるタスクのみ |

補足: タスク更新・削除は単純なロール判定ではなく、作成者・担当者・チーム管理者条件を組み合わせる。

### 5.3 コメント系マトリクス

| 機能 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|
| コメント一覧取得 | ○ | ○ | タスク参照権限あり |
| コメント投稿 | ○ | ○ | タスク参照権限あり |
| コメント更新 | △ | ○ | 投稿者本人のみ、または ADMIN |
| コメント削除 | △ | ○ | 投稿者本人のみ、または ADMIN |

補足: コメント更新・削除は「自分のコメントのみ編集・削除」という業務ルールを反映する。

### 5.4 添付ファイル系マトリクス

| 機能 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|
| 添付一覧取得 | ○ | ○ | タスク参照権限あり |
| 添付アップロード | ○ | ○ | タスク更新権限あり |
| 添付ダウンロード | ○ | ○ | タスク参照権限あり |
| 添付削除 | △ | ○ | 添付登録者本人または ADMIN |

### 5.5 チーム系マトリクス

| 機能 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|
| チーム一覧取得 | ○ | ○ | 自分の所属チームのみ |
| チーム詳細取得 | ○ | ○ | 所属チームのみ |
| チーム作成 | ○ | ○ | ログイン済みユーザー |
| メンバー一覧取得 | ○ | ○ | 所属チームのみ |
| メンバー追加 | - | ○ | チーム管理者のみ |
| メンバー削除 | - | ○ | チーム管理者のみ |
| ロール変更 | - | ○ | チーム管理者のみ |

### 5.6 通知・履歴系マトリクス

| 機能 | MEMBER | ADMIN | 条件 / 補足 |
|---|---|---|---|
| 通知一覧取得 | ○ | ○ | 自分宛のみ |
| アクティビティログ参照 | △ | ○ | MEMBER は自分関連のみ、ADMIN はチーム内参照可 |

## 6. 代表的な認可ルール

- タスク更新権限：タスク作成者、タスク担当者、または所属チームの管理者のみ更新可能
- タスク削除権限：タスク作成者、または所属チームの管理者のみ削除可能
- コメント更新・削除権限：コメント投稿者本人、または所属チームの管理者のみ実行可能
- メンバー管理権限：対象チームにおける ADMIN ロール保有者のみ実行可能

## 7. 実装方針

| レイヤ | 役割 | 備考 |
|---|---|---|
| Controller | 認証済みユーザー情報の取得、Service への委譲 | 認可判定ロジックは持たない |
| Service | 対象データ取得、所有者確認、所属確認、ロール確認 | 認可エラー時は例外送出 |
| Repository | DBアクセスのみ | 認可判定ロジックは持たない |

## 8. エラー設計との対応

本システムの認可エラーは、エラー設計書で定義した HTTP ステータスおよびエラーコード体系に従って返却する。

| エラーコード | HTTP | 意味 | 主な発生条件 |
|---|---:|---|---|
| ERR-AUTH-005 | 403 | 操作権限がありません | 共通的な権限不足 |
| ERR-TASK-005 | 403 | タスク更新権限がありません | タスク更新時、作成者または担当者以外 |
| ERR-TASK-006 | 403 | タスク削除権限がありません | タスク削除時、作成者以外 |

### 8.1 現行実装との対応

| 機能 | 返却コード | HTTP | 補足 |
|---|---|---:|---|
| 共通権限制御 | ERR-AUTH-005 | 403 | Spring Security の AccessDeniedException など |
| タスク更新 | ERR-TASK-005 | 403 | Service 層の更新権限判定で返却 |
| タスク削除 | ERR-TASK-006 | 403 | Service 層の削除権限判定で返却 |

### 8.2 補足

- 認可エラーはすべて `403 Forbidden` として返却する
- レスポンス形式はエラー設計書で定義した `ErrorResponse` に統一する
- 認証エラー（未ログイン、JWT 不正、期限切れ）は認可エラーとは分けて扱い、`401 Unauthorized` を返却する
- コメント系、添付ファイル系、チーム管理系の権限エラーコードは、将来機能追加時にエラー設計書へ合わせて追加定義する


## 9. 設計意図

本システムでは、JWT による認証と、ロールおよびリソース所有者条件による認可を分離して設計している。  
単純なロールベース制御だけでは、コメント編集や添付削除のような「本人のみ許可」の条件を表現しきれないため、RBAC に加えて所有者ベースの制御を組み合わせた。  
これにより、チーム管理機能のような管理権限と、コメント編集のような本人権限を両立できる構成としている。

## 10. 面接での説明ポイント

- 認証と認可を分離している
- 認可は単純なロールだけでなく所有者条件も含めている
- コメント編集 / 削除は本人のみという実務的なルールを入れている
- チーム管理系は ADMIN ロールで制御している
- 判定ロジックは Service 層に寄せている
